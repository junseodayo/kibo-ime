# 물리 키보드 3개국어 입력기(IME) — 기능 정의서

- **문서 상태:** 초안 (개인용 MVP 우선 · 추후 출시 검토)
- **대상 기기:** Unihertz Titan 2 (Android 15) — 추후 BlackBerry 등 물리 키보드 기기로 확장 고려
- **상태 표기:** `[확정]` 합의됨 / `[추후]` 차후 단계 / `[확인]` 구현 전 검증 필요

---

## 0. 개요 및 설계 원칙

- 한국어 · 영어 · 일본어를 입력하는 **물리 키보드 전용 입력기(IME)**. 시중 키보드들이 일본어를 잘 안 만들어주는 공백을 메우는 것이 핵심 동기.
- **물리 키보드가 1차 입력 수단**, 온스크린 키보드는 비상용 폴백.
- **기기 비종속 설계** — 타이탄2 전용 가정(전용 트랙패드, 전용 버튼 등)을 코드에 박지 않고, 일반 안드로이드 키 처리 위에 구현. `[확정]`
- 현재는 개인용이지만 안정화 시 출시 가능성을 열어둠 → 출시 시 살아나는 항목은 §16에 별도 정리.

---

## 1. 아키텍처 개요

- 안드로이드 **`InputMethodService` 기반 IME**.
- 소프트 키보드 렌더링뿐 아니라 **하드웨어 키 이벤트(`onKeyDown`/`onKeyUp`, Key Character Map)를 가로채서** 언어 엔진으로 라우팅. ← 일반 키보드 앱과 가장 다른 지점.
- **입력 디스패처:** 들어온 키를 글자키 / 수정자(shift·sym·fn·alt) / 레이어 입력으로 구분해 현재 언어 엔진에 분배.
- 타이탄의 **정전식 트랙패드 제스처·전용 프로그래머블 버튼은 직접 구현하지 않고 OS/스톡 펌웨어에 위임.** `[확정]`

---

## 2. 언어 모드 상태기계 `[확정]`

- **언어:** 한국어 → English → 日本語 (기본 순서).
- 순서는 **설정에서 드래그로 변경** 가능.
- **언어 전환키:** 사용자가 직접 키/조합을 눌러 지정(press-to-assign).
  - 기본값 **fn + space** (타이탄2에 fn 키 존재 확인됨).
  - **최소 1개 수정자 포함** 요구(단일 글자키 금지 → 타이핑 방해 방지).
  - **충돌 감지:** 지정 시 그 키 이벤트가 IME에 도달하면 등록(기존 바인딩과 중복 검사). 도달하지 않으면 OS가 선점한 조합 → "이 조합은 시스템이 사용 중. 다른 조합을 누르세요" 안내.
  - 한계: IME는 OS가 넘겨주는 이벤트만 봄 → 시스템 선점 조합은 강제로 뺏어 쓸 수 없음.
- **모드 인디케이터:** 현재 언어/대소문자/레이어를 항상 알 수 있는 신호 필수(툴바에 표시). 시각 디자인은 추후.

---

## 3. 한국어 엔진 `[확정]`

- **두벌식 표준** 매핑. (추후 입력 방식 변경 옵션 — `[추후]`)
- **한글 오토마타:** 자모 조합, 받침/겹받침, 도깨비불 처리.
- **백스페이스:**
  - 짧게 = **자모 단위 삭제** (조합 중인 글자).
  - 길게(꾹) = **단어 단위 삭제** (이미 확정된 텍스트, 공백 기준).
  - *조합 중 글자*와 *확정된 텍스트*의 경계를 명확히 정의해 동작 분기.

---

## 4. 영어 엔진 `[확정]`

- 패스스루 + Shift / Caps Lock.
- **자동완성 · 자동교정 없음.**

---

## 5. 일본어 엔진 `[확정]` — 본 앱의 핵심 차별점

- 입력 흐름: **로마자 → 가나 → 한자 변환 → 후보 선택 → 확정.**
- **변환 엔진: Mozc 기반** (대규모 사전 확보 목적). → 안드로이드 NDK 빌드가 본 프로젝트 최대 난구간. `[확인]`
- **사전 보강:** 인명·지명·역명 + 유행어/신조어를 **NEologd 계열 사전**에서 가져와 통합.
  - 유행어는 시간이 지나면 낡으므로 **사전 갱신이 가능한 구조**로 설계.
  - 사전 라이선스는 출시 시 점검 대상(§16).
- **후보 조작:** 터치 위주. **Space = 다음 후보, Enter = 확정.** 숫자키 직접 선택 없음.
- **추가 처리:** 가타카나 변환 키, 전각/반각(全角·半角) 처리 포함.

## 6. 숫자/심볼 레이어 — 정확성 요구사항 `[확정]`

물리 키보드는 숫자 전용 행이 없고 **`alt`/`sym` 레이어**로 숫자·기호를 입력(키에 작게 인쇄된 문자). 자주 쓰는 경로라 조합을 깨지 않는 것이 핵심.

- **OS의 Key Character Map을 활용** — `getUnicodeChar(metaState)`로 OS가 해석해준 문자를 그대로 받음. **alt/sym 매핑을 직접 하드코딩하지 않음** → 기기 비종속 유지.
- 처리 규칙:
  - alt/sym 레이어 입력이면 → **언어 엔진에 먹이지 않고**, 진행 중인 조합을 **먼저 확정한 뒤** 해당 문자를 삽입.
  - 평범한 글자키면 → 현재 언어 엔진으로.
- 한·영·일 **모든 모드에서 동일하게 정확**해야 함. 일본어 모드는 전각/반각 규칙 적용.

## 8. 클립보드 `[확정]`

두 영역을 분리 제공, 입력창에 바로 붙여넣기 가능.

- **① 복사 히스토리:** 복사한 텍스트 목록.
- **② 텍스트 프리셋:** 자주 붙여넣는 문구를 미리 등록한 목록.
- 결정 사항:
  - 히스토리 **최대 개수**(예: 50) + **중복 제거**.
  - 자주 쓰는 항목 **고정(pin)** 가능.
  - **비밀번호 입력칸은 히스토리에서 제외.**
  - 클립보드 히스토리는 **백업/동기화 대상에서 제외**(민감·휘발성).

## 10. 온스크린 키보드 (토글) `[확정]`

- **툴바 토글 버튼**으로 켜고 끄기.
- 심볼/이모지는 이 토글과 별개의 툴바 버튼.

---

## 11. 앱별 기본 언어 `[확정]`

- **마스터 토글(ON/OFF)** — 원치 않는 사용자를 위해 끌 수 있음.
- **OFF:** 전역 "직전 사용 언어" 유지(앱 전환과 무관).
- **ON:** 앱 진입 시 **앱별 언어 매핑** 조회.
  - 매핑은 키보드 설정에서 패키지명 기준으로 지정(앱 감지는 `EditorInfo.packageName`로 신뢰성 확보).
  - **매핑 없는 앱은 직전 사용 언어로 폴백.**
- 선택 옵션: 수동 매핑 대신 **"앱별 마지막 사용 언어 자동 기억"(auto-learn)** 방식도 가능. 일단 수동 매핑을 기본으로. `[추후]`

## 12. 백업 / 복원 `[추후]`

비용·프라이버시 부담 순 단계. 클립보드 히스토리는 동기화 제외.

| 단계 | 방식 | 특징 |
|---|---|---|
| 0 | 안드로이드 자동 백업 | 코드 거의 0, 같은 구글 계정 재설치 시 자동 복원 |
| 1 | JSON 내보내기/가져오기 | 오프라인·비공개. **MVP 권장** |
| 2 | 사용자 본인 클라우드(드라이브 등) | 다기기 + 별도 계정 불필요 + 데이터는 사용자 소유. **다기기 필요 시 권장** |
| 3 | 자체 계정 연동(서버 운영) | 실시간 다기기. 서버·인증·프라이버시 책임 큼. 명분 확실할 때만 |

→ 지금 구현 안 함. 추후 필요 시 **3단계로 바로 가지 말고 2단계 우선 검토.**

## 14. 기기 비종속 원칙 `[확정]`

- 타이탄 전용 키(트랙패드 제스처, 전용 버튼, fn 존재)를 **당연하게 가정하지 않음.**

- 기기별 키 차이는 **"키 직접 지정"(§2)** 으로 흡수.

## 17. 개발 단계 로드맵 (제안)

- **Phase 0 — 기반:** IME 골격, 입력 디스패처, 언어 상태기계, 모드 인디케이터, 키 직접 지정, 숫자/심볼 처리.
- **Phase 1 — 기본 언어:** 한국어 오토마타 + 영어 엔진.
- **Phase 2 — 일본어:** Mozc 통합 + 후보 흐름 + 사전 보강 (핵심 차별점, 최대 작업량).
- **Phase 3 — 편의:** 사용자 사전, 클립보드(히스토리+프리셋), 앱별 언어.
- **Phase 4 — 마감/출시 준비:** 온스크린 키보드 토글, 백업/복원, UI 테마, 출시 점검(§16).
